# 🍳 최종 기획서: "레시피메이트" - 레시피 기반 식재료 공동구매 플랫폼

## 📋 핵심 컨셉

### 🎯 왜 이 컨셉인가?

1. **차별화 명확**: 일반 공동구매 ❌ → 레시피 중심 식재료 공구 ✅
2. **API 연동 자연스러움**: 레시피 데이터 → 재료 추출 → 공구 생성 흐름이 명확
3. **기능 확장 용이**: 하나의 컨셉(요리)에서 여러 기능 파생 가능
4. **타겟 적합**: 대학생(자취생)의 실제 니즈 - 배달비 절약, 식재료 낭비 방지
5. **구현 난이도 적절**: 초보자도 가능하면서 완결성 확보

## 🌟 프로젝트 개요

**프로젝트명**: 레시피메이트 (Recipe Mate)

**부제**: 레시피로 시작하는 똑똑한 식재료 공동구매

**핵심 가치**

- "혼자 사기엔 부담스러운 식재료, 함께 사요!"
- "요리 레시피를 보고 → 필요한 재료만 → 공동구매"

**타겟 사용자**

- 1차: 자취하는 대학생 (기숙사생, 자취생)
- 2차: 1인 가구, 요리 초보자

## 🔌 핵심 API 선정 (3개)

### ✅ 1. TheMealDB API (필수 - 레시피 데이터)

**용도**: 레시피 검색, 상세 정보, 재료 리스트 제공

**비용**: 개발용 무료 (키="1")

**활용 기능**:

- 레시피 검색 (`search.php?s=이름`)
- 레시피 상세 조회 (`lookup.php?i=ID`)
- 재료별 레시피 필터 (`filter.php?i=재료`)
- 랜덤 레시피 추천 (`random.php`)
- 카테고리별 조회 (`categories.php`)

**데이터 구조**:

- `strMeal` (레시피명)
- `strIngredient1~20` (재료 목록)
- `strMeasure1~20` (분량)
- `strInstructions` (조리법)
- `strMealThumb` (이미지)
- `strCategory` (카테고리)

### ✅ 2. 식품안전나라 COOKRCP01 API (준필수 - 한국 레시피)

**용도**: 한국 음식 레시피 + 영양 정보

**비용**: 무료 (인증키 발급 필요)

**활용 기능**:

- 한식 레시피 700+ 제공
- 영양 정보 (칼로리, 단백질, 탄수화물 등)
- 조리 단계별 이미지
- 레시피명, 재료 정보 제공

**장점**: TheMealDB(영어)를 보완, 한국 사용자 경험 강화

### 🔄 3. Google Cloud Translation API (선택 - 번역)

**용도**: TheMealDB 영어 데이터 → 한국어 자동 번역

**비용**: 월 50만 문자 무료

**활용 시점**: 5주차 이후 여유 있으면 추가

### 🎬 사용자 액션 흐름

### 1) 레시피 검색

- **입력**: 검색창에 키워드 입력
    - 예: "파스타", "닭가슴살", "간단한 요리"
- **검색 방식**:
    - 레시피명 검색: TheMealDB `search.php?s=파스타`
    - 재료명 검색: TheMealDB `filter.php?i=chicken`
    - 한글 검색 시: 식품안전나라 API 병행 검색
- **결과 표시**: 일치하는 레시피 카드 목록
- **결과 없음**: "검색 결과가 없습니다" 메시지 + 랜덤 추천 3개

### 2) 카테고리 필터 선택

- **버튼 클릭**: [한식], [양식], [중식], [일식], [디저트]
- **동작**:
    - TheMealDB: `categories.php` 활용
    - 식품안전나라: 해당 카테고리만 필터링
- **UI 변화**: 선택된 버튼 하이라이트 (색상 변경)
- **중복 선택**: 불가능 (라디오 버튼 방식)

### 3) 정렬 변경

- **드롭다운 선택**:
    - 최신순: 최근 추가된 레시피 (캐시 시점 기준)
    - 인기순: 해당 레시피 기반 공구 개수 많은 순
    - 간단한순: 재료 개수 적은 순 (strIngredient 카운트)
    - 이름순: 가나다/ABC 순
- **즉시 반영**: 선택 즉시 목록 재정렬

### 4) 레시피 카드 클릭

- **동작**: 카드 전체 영역 클릭
- **이동**: 레시피 상세 페이지로 이동
- **카드 정보**:
    - 대표 이미지 (strMealThumb)
    - 레시피명 (strMeal 또는 번역된 한글명)
    - 난이도 (별점, 재료 개수로 추정: 5개 이하=하, 10개 이하=중, 그 이상=상)
    - 예상 조리 시간 (기본값 표시, API에 없으면 "약 30분")
    - 🍳 아이콘 (레시피 연동 표시)

### 5) 더보기/페이지네이션

- **더보기 버튼**: 클릭 시 다음 12개 레시피 추가 로드 (무한 스크롤 방식)
- **페이지네이션**: 번호 클릭 시 해당 페이지로 이동 (한 페이지 12개)

### 🎬 사용자 액션 흐름

### 1) 재료 체크박스 조작

- **기본**: 모든 재료 체크됨 (☑️)
- **클릭**: 체크 해제 (☐) → 이미 있는 재료 제외 가능
- **용도**: 공구 만들기 시 체크된 재료만 포함
- **시각적 피드백**: 체크 해제된 항목은 흐리게 표시

### 2) 인원 설정 조정

- **[+] 버튼**: 클릭 시 인원 1명 증가
- **[-] 버튼**: 클릭 시 인원 1명 감소 (최소 1명)
- **숫자 직접 입력**: 숫자 영역 클릭 → 키보드 입력 가능
- **실시간 계산**: 인원 변경 시 재료 분량 자동 재계산
    - 예: 4인분 → 10인분 선택 시
    - "스파게티 면 400g" → "스파게티 면 1000g" 자동 변환

### 3) 진행 중인 공구 확인

- **표시 조건**: 이 레시피 기반으로 만들어진 모집중 공구 있으면 표시
- **[공구 상세 보기] 클릭**: 해당 공구 상세 페이지로 이동
- **없으면**: 이 섹션 자체가 숨김

### 4) [이 재료로 공구 만들기] 버튼 클릭 ⭐핵심⭐

- **로그인 확인**:
    - 비로그인 → 로그인 페이지로 리다이렉트
    - 로그인됨 → 공구 작성 페이지로 이동
- **데이터 전달**:
    - 레시피 ID
    - 체크된 재료 목록 (strIngredient)
    - 분량 목록 (strMeasure, 인원 설정 반영)
    - 레시피명
    - 레시피 이미지
- **자동 입력**: 공구 작성 페이지에서 아래 항목 자동 채워짐

        - 진행률 실시간 업데이트 (3명 → 4명)
        - 진행률 바 색상 변화 (60% → 80%)

- **찜 해제**:
    - 꽉 찬 하트 ♥ → 빈 하트 ♡
    - "찜 목록에서 제거되었습니다" 토스트 알림

- **성공**:

    - 자동 로그인 + 홈으로 이동

```
1. 로그인 상태에서 [레시피 탐색] 메뉴 클릭

   ↓

   ↓

   ↓

   ↓
6. 이미 있는 재료 체크 해제 (예: 쌀, 소금)
   ↓
7. [이 재료로 공구 만들기] 버튼 클릭
   ↓
8. 공구 만들기 페이지로 이동 (자동 입력됨)

   ↓

   ↓
11. 공구 등록 완료! 공구 상세로 이동
   ↓
12. 마이페이지 > 내 공구에서 확인
```

## 📝 13. API 연동 상세

### TheMealDB API 주요 엔드포인트

```jsx
// 1. 레시피 검색 (이름)
GET <https://www.themealdb.com/api/json/v1/1/search.php?s=파스타>
→ 응답: { meals: [ {...레시피 객체...} ] }

// 2. 레시피 상세 (ID)
GET <https://www.themealdb.com/api/json/v1/1/lookup.php?i=52772>
→ 응답: { meals: [ {...상세 레시피...} ] }

// 3. 재료별 레시피 필터

// 4. 랜덤 레시피

→ 응답: { meals: [ {...랜덤 레시피...} ] }
```

## "레시피메이트" 기획서 - 보완 및 제안 사항 (Areas for Improvement)

기획서가 이미 매우 훌륭하지만, 실제 개발 및 운영 관점에서 다음 몇 가지를 추가로 고려하면 더욱 완성도를 높일 수 있습니다.

1. **재료 데이터의 표준화 및 매핑 문제**:
    - TheMealDB의 `strIngredient`와 식품안전나라의 '재료정보'는 자유 형식의 텍스트입니다. 예를 들어, 같은 '달걀'이라도 'egg', 'eggs', '계란', '달걀(1개)' 등 다양하게 표현될 수 있습니다.
    - **제안**: '레시피 기반 공구 만들기'의 핵심은 여러 레시피의 재료를 하나의 공구로 묶는 것인데, 이 문제를 해결하지 않으면 기능 구현이 어렵습니다. 따라서 **'재료 마스터 데이터'** 또는 **'재료 동의어 사전'**을 내부 DB에 구축하는 과정이 필요합니다. 개발 초기 단계(Week 1-2)에 이 데이터 구조 설계 및 기본 데이터 구축 계획을 추가하는 것을 권장합니다.
2. **가격 책정의 현실성 문제**:
    - '레시피 기반 공구 만들기' 시 사용자가 직접 '총 금액'을 입력하게 되어 있습니다. 하지만 다양한 재료의 현재 시장 가격을 개인이 정확히 알기는 어렵습니다.
    - **제안**: 초기 버전에서는 현재 방식(사용자 직접 입력)을 유지하되, **'보너스 아이디어'**나 **'향후 개선 과제'**로 **'주요 식재료 평균 가격 정보 제공'** 기능을 추가하면 좋겠습니다. (예: 별도의 크롤링이나 공공데이터 API를 활용) 이는 사용자가 더 합리적인 가격으로 공구를 설정하는 데 도움을 줄 것입니다.
3. **'매너온도' 시스템의 구체화**:
    - 현재 후기 별점에 따라 매너온도가 변동되는 로직은 좋습니다. 다만, 분쟁 발생 시나 비정상적인 평가(일명 '평점 테러')에 대한 대응책이 필요합니다.
    - **제안**: **'신고 기능'**을 추가하고, '누적 신고 횟수에 따른 매너온도 차감' 로직을 명시하면 더 신뢰도 높은 시스템을 구축할 수 있습니다. 마이페이지 설정 탭의 '계정 탈퇴'처럼, 운영 정책을 조금 더 구체화하는 것이 좋습니다.
4. **기술 스택 관련**:
    - 프론트엔드에서 Vanilla JavaScript를 사용하는 것은 좋지만, Thymeleaf와 함께 사용할 경우 동적인 UI 변경(예: 인원수 변경 시 가격 자동 계산, 실시간 진행률 업데이트)이 다소 번거로울 수 있습니다.
    - **제안**: 복잡한 UI 변경이 예상되는 '공구 만들기', '공구 상세' 페이지 등 특정 페이지에 한해 **Alpine.js**나 **htmx** 같은 가벼운 라이브러리를 함께 사용하는 것을 고려해볼 수 있습니다. 이는 개발 생산성을 높여줄 수 있습니다. (물론, 팀의 학습 곡선을 고려하여 신중히 결정해야 합니다.)
